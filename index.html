设计理念

AI 是助手，不是法官

我们不认为 AI 能替代律师。法典的定位是法务工作的效率放大器——把翻合同、查法条、标风险的体力活交给 AI，把判断和决策留给人类。

01

只标注，不下结论

识别风险条款、附上法律依据，但不替你说"这个合同能签"或"这个官司能赢"。

02

多角色协作

不是一个大模型包办一切。审查官、研究员、合规顾问——各司其职、交叉验证，像真实团队一样工作。

03

安全边界内置

每个角色的 SOUL.md 中都写死了安全规则，每次输出强制附带免责声明。不是事后补充，是出厂设置。

场景：员工王五因绩效考核被解雇，公司未付经济补偿
裁判规则（指导案例18号）
「等级考核中居于末位，不等同于"不能胜任工作"」——单独考核不合格≠合法解除。公司须先履行培训或调岗前置程序。
赔偿金还是补偿金？
缺少任一程序=违法解除→赔偿金2N（法87条）。王五案：公司仅凭考核辞退，典型违法，应主张2N。
类案参考
陕西(2024)案：类似科技行业3年工龄，法院判赔偿金97,395元——培训内容须与岗位相关，走过场式培训无效。
用人单位需同时证明四项，缺一不可：
① 考核制度经民主程序制定 → ② 考核结果已告知劳动者 → ③ 已履行培训或调岗义务（培训须与岗位相关、时长合理） → ④ 培训/调岗后仍不能胜任。仅凭绩效考核结果辞退 = 违法解除。

选题策略

为什么是「AI 法务」而非「AI 律师」？

这个定位不是在回避挑战，而是在回应法学界对 AI 的核心质疑。

🏛️ 法学界的三重担忧
① 机械断案
法律是人的判断，同一案件在不同法官手里有不同结果。AI 无法处理个案特殊性。
② 可追责性
如果 AI 给出错误建议并造成损失，谁来负责？现行法律体系没有 AI 的位置。
③ 算法黑箱
深度学习模型的决策过程不可解释，在法律行业尤其不可接受——判决必须说理。
🛡️ 我们的回应
不替代人，帮人省力
AI 负责标注风险点、检索法规、整理判例——这些是可标准化的机械劳动。法律判断永远留给执业律师。
标注来源，可追溯
每条法条引用标注编号，每条建议提供法律依据。不编造、不揣测、不确定的宁可不写。
安全边界内置，不可绕过
每个 Agent 的 System Prompt 都写死了安全规则——不下结论、必带免责声明、不处理真实个人信息。
安全边界

知道什么不能做，比知道能做什么更重要

我们认真想过 AI 在法律领域的边界。这批 Agent 不是用来替代律师的——它们是律师的放大镜和速记员。

🛡️

内置安全规则（不可绕过）

以下规则写入每个 Agent 的 SOUL.md 核心层，不是可选项，是出厂设定：

01 不下法律结论——只标注风险、提供参考，不替人判断"该不该签""能不能赢"
02 每条产出强制附带免责声明——"不构成法律意见，请由执业律师确认"
03 不编造法条——不确定的立法判例宁可不写，不可伪造
04 涉及刑事/人身安全/重大财产——强制提示必须咨询专业律师
05 不处理未经脱敏的真实个人信息——如发现自动提醒使用者
06 总指挥验收机制——任何子Agent产出必须通过总指挥核对后才交付
这些规则是怎么写进每个角色的 System Prompt 的：
# 合同审查官 SOUL.md 节选

安全边界：
1. 不在能力范围内的领域明确说"建议检索专业法律数据库"
2. 每条产出结尾：⚠️ 以上为AI辅助分析，不构成法律意见，请由执业律师确认。
3. 涉及重大财产权益的条款 → 加粗提示"强烈建议咨询专业律师"
4. 不确定的宁可不写，不编造法条
# 总指挥 SOUL.md 节选

安全边界（最关键的原则）
1. 不下法律结论
2. 不确定的地方标注"建议人工核实"
3. 每条产出结尾附免责声明
4. 涉及刑事/人身安全/重大财产 → 明确告知必须咨询专业律师
# 法规研究员 SOUL.md 节选

不提供"结论性意见"——只提供条文和案例，
由合规顾问或总指挥综合判断。
不假装知道所有法律——不在能力范围的明确说"建议检索专业法律数据库"
# 文书起草员 SOUL.md 节选

起草的文书是模板性质，使用时需根据实际情况调整。
不代签、不代发——只输出文书。
涉及重大权益的文书加注"建议由执业律师审阅后使用"。 不伪造证据、不虚构事实。
危机应对

如果 AI 出错了怎么办？

这是每个 AI 法律产品必须回答的问题。我们不回避。

可能的出错方式
法条编号编造。AI 可能生成听起来合理的但不存在的法条编号（如"合同法第12条"）。我们在角色指令中要求"不确定的法条不写"，并由总指挥二次核对。
风险漏检。AI 可能遗漏某些不常见的风险点。降低此风险的方式是多角色交叉——合规顾问的二次审查是专门用来兜底的。
案例不可靠。AI 编造的真实感案例最难辨别。我们要求案例分析官只引用有可靠来源的案例，不确定来源的标注"待核实"。
三重防御机制
第一层：角色内约束。每个 Agent 的 System Prompt 都写死了"不确定时不写、不编造法条、不预测判决"。如果模型试图突破，这些指令会在语义层被优先执行。
第二层：多角色交叉验证。审查官的发现由研究员核査法条、由合规顾问二次分析。三个角色独立运行同一个问题，单点故障概率大幅降低。
第三层：人力终审（设计如此）。这是最重要的一层。AI 的产出从设计上就不用作终审——每条都有免责声明，设计理念就是"AI 标注风险，律师做判断"。错误 AI 标注的风险点，律师可以纠正；未检测到的风险点，律师可以发现。AI 的角色是放大镜，不是裁判。
我们的立场：我们不声称 AI 不会出错。我们声称的是——当它出错时，伤害被控制在了"可纠正的参考建议"范围内，永远不会孤军深入替代法律判断。
技术实现

不是黑箱，七套 System Prompt

每个 Agent 的背后不是魔法——是一份精确到文字、人手可读的 System Prompt（SOUL.md）。安全边界不是"事后审查"，是编译在指令里的出厂设置。

🧱
纯 Prompt 驱动
无需微调、无需 RAG 系统、无需向量数据库。7个角色的全部能力定义在 SOUL.md 中——每个角色约 2-4KB 的纯文本，人手可读、可审计、可修改。
📐
结构化输出约定
每个角色定义了自己的输出格式——合同审查官输出"审查清单"（🔴🟡🟢标注），法规研究员输出"法条检索报告"（编号+原文+适用说明），合规顾问输出"合规分析报告"（排序+策略）。
🔁
串行 Pipeline
审查官输出 → 研究员输入 → 研究员输出 → 合规顾问输入 → 合规报告。每个环节的上游结果作为下游任务的上下文嵌入，无需人工中转，全自动调度。
🔄
并行子任务
总指挥可同时向多个 Agent 派发独立任务（如合同审查官+案例分析官并行运行），互不阻塞。当前实现中，三个 Demo 合同的审查是同时启动的。
📎
上下文传递
上游 Agent 的完整输出（含风险标注、法条引用）作为下游的 task 上下文传递。模型自动提取相关信息，无需显式解析或结构化中间层。
💸
极低成本
全部 Agent 使用 DeepSeek V4 Pro（¥2/百万 token 出）。单次完整 Pipeline（3角色×14,000 tokens）总成本 ¥0.03。与人工审查 ¥500-2,000 相比，成本降低 4-5 个数量级。
